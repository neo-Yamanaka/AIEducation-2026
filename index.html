Generative AI  ×  Claude Code

AIと、 賢く付き合う技術。

生成AIは一瞬でコードを生む。けれど、前提を間違えたまま走ることもある。このサイトは、品質を保ちながらAIと開発を進めるための「ルール」と「実践テクニック」を、用語解説つきで一気に学べる場所です。

00 — Foundation

まず、土台の3つ。

細かいルールに入る前に、この3語だけ押さえておけば全体が読めます。下線つきの用語は、ホバー（スマホはタップ）で意味が出ます。

i.

生成AIとは

文章やコードを自動で作り出すAI。プロンプト（指示文）を与えると、それに沿った出力を返してくれます。ChatGPT や Claude がその代表です。

ii.

AI駆動開発とは

生成AIに実装や調査・レビューを任せながら進める開発スタイル。速い反面、「どこまで任せ、どこを人が確認するか」の設計が品質を分けます。

iii.

Claude Codeとは

Anthropic の Claude を、コマンド操作でコーディングに使えるツール。ルールファイルやスラッシュコマンドで、AIに前提を毎回読ませて作業させられます。

Part Ⅰ — Rules

守るべきこと（1〜6章）

ツールに依存しない、普遍的な運用原則。チームで共有する前提として読んでください。

01

ルール・方針の統一

Unify the rules

AIを使う開発では、人ごとに指示の出し方が違うと、コード品質や設計がそろわなくなります。チームであらかじめ共通ルールを決め、AIにも同じ前提で作業させます。

明文化したいコーディングルール

  • 命名規則 / ディレクトリ構成（＝フォルダの置き方）
  • コメントの書き方 / エラーハンドリング方針
  • テスト方針、そして「誰がどのファイルを触るか」の担当範囲

AIへの前提指示（設計方針）

  • 勝手に既存仕様を変えない／推測だけで実装しない
  • 不明点は確認する／影響範囲を調べてから直す
  • 指示された範囲外を直さない／大きな変更は実装前に方針を提示する
Whyこうした前提を先に渡すことで、AIの「やりすぎた修正」や意図しない変更のリスクを下げられます。
02

ルールの集約と共有

Centralize into rule files

1章で決めたルールは、持っているだけでは効きません。ルールファイルとして、AIが毎回自動で読み込む場所に置いて初めて実効性を持ちます。

代表的なルールファイル

  • CLAUDE.md（Claude Code）／ AGENTS.md ／ .cursorrules（Cursor）
  • ドメイン固有のルールは .claude/rules/*.md のように、特定パスを触ったときだけ読ませる

運用のコツ

  • ファイルはシンプルに保つ。詰め込みすぎると毎回の読み込みトークンが増え、肝心のルールが埋もれる
  • 全体に効かせたいルールは先頭に、特定分野のルールは分離する
  • 重要な方針は「レビュー時に必ず確認する観点」としても書いておくと、節目ごとにチェックされてブレにくい
03

セキュリティ・機密情報の取り扱い

Security & secrets

AIにコードやデータを渡す開発では、安全面のルールを最初に決めます。特に個人情報や認証情報を扱うシステムでは重要度が高いです。

  • 機密情報（APIキー・.env・本番データ等）をAIのコンテキストに渡さない
  • プロンプトインジェクションに注意。外部から取り込んだ文章は「データ」として扱わせる
  • コマンド自動実行は許可/拒否リストで制御。rm・git push・.env読み取り等はAIによる自動実行を拒否し、必要なときは人が内容を確認してから実行する
  • AIが勧めるライブラリは実在する正規品か確認（パッケージハルシネーションはサプライチェーン上のリスク）
Caution「AIが勝手に破壊的な操作をした」という事故の大半は、権限と機密情報の扱いをルール化していないことが原因です。
04

データベース設計・変更管理

Database changes

AIが「必要だ」と判断して勝手にテーブルを増減したり、スキーマを変えたりすると、既存機能や他メンバーの作業に大きく影響します。DB設計は後戻りが難しいぶん、慎重に。

  • テーブルの追加・削除は事前承認制にする
  • カラムの追加・変更・削除はレビュー対象にする
  • マイグレーションファイルの内容を必ず確認する
  • 既存データへの影響を事前に確認／複数人開発では承認フローを設ける
PrincipleAIの提案をそのまま採用せず、人間による確認を必須に。
05

ソース修正をAIに任せる際の作業ルール

Editing workflow

修正前後の基本フロー

修正前…対象の仕様・既存コードの構造・影響範囲をAIに確認させる。
修正後…テスト実行・差分確認・不要な変更が無いかをチェック。

  • 頼んでいないリファクタリングをさせない
  • 実装内容をAIに説明させ、作業者自身が理解する
  • タスクごとに完了の定義（DoD）を先に決める

テストの健全性を守る

AIはテストを通すために、テスト自体を弱めたり消したり、自明にパスする内容へ書き換えることがあります。

  • 既存テストを勝手に変更・削除させない
  • バグ修正は、先に「失敗するテスト」を書かせてから直させる
  • リグレッションテストを残し、何を保証しているか把握する

バージョン管理・PR運用

  • タスク単位でブランチを切り、コミットは小さく
  • AIに自動コミットやforce-pushをさせない
  • レビューはPRの差分ベースで行う
Nextこの「確認してから直す」流れは、実践編の A「意図を固める」（プラン承認・計画モード） でさらに具体化できます。
06

AI出力との向き合い方

How to treat AI output

AIを使うと、内容を理解しないまま開発を進められてしまう落とし穴があります。生成コードをそのまま使うと、問題発生時の原因調査や修正が難しくなります。

  • AIに実装内容を説明させ、自分で理解してからマージする
  • 重要・複雑な処理は人によるレビューを必須にする
  • ハルシネーションに注意。見覚えのないAPIや設定は公式ドキュメントで裏取り（MCP等で最新仕様を参照させる）
In shortAIの出力は「たたき台／補助的な提案」。最終判断は人間が行う、を前提に。下の信頼度メーターで体感し、実践編の B「品質を高める」 で裏取りの具体策（MCP等）へつなげましょう。

どこまで信用する？

AI開発で最も難しい問いのひとつ。スライダーを動かして、信頼度の「ちょうどいい」を体感してください。

Part Ⅱ — Practice

実践テクニック（4つの視点）

日々の開発を効率化・高品質化する具体策を、意図を固める・品質を高める・スケールさせる・コストを抑えるの4つの視点で整理しました。Claude Code での開発をベースに説明していきます。

A

意図を固める

Align intent first

AI開発で最も手戻りが出るのは「頼んだつもり」と「作られたもの」のズレ。実装に入る前に、方針・範囲・手順を握っておくのが、いちばん効きます。

プラン承認 ＆ 計画モード

いきなり実装させず、先に方針・影響範囲・手順を提示させ、人が承認してから着手させます。計画モードを使うと、AIは読み取り・調査だけを行い、ファイルを書き換えずに計画を立てます。承認するまで変更が走らないので、意図ズレを未然に防げます。

Why先に計画を挟むと、間違った方向へ大量のコードが生成される事故を防げます。仕様を先に固める spec駆動 も同じ発想で、手戻りと無駄なトークン消費を減らせます。

対話で要件を掘り下げる

重要な作業ほど、いきなり作らせず対話で要件を詰めます。次の一文を添えるだけでも精度が変わります。

  • 「推測での実装を避け、不明点は先に質問してください」
  • 「実装の前に、方針と影響範囲を箇条書きで見せてください」
  • 「この理解で合っているか、作業前に確認させてください」

止める・戻す — 軌道修正の基本

「間違っても止めて戻せる」と知っておくと、安心して任せられます。

  • 方向がズレたと感じたら ESC でいつでも中断し、指示を出し直す（最後まで待つ必要はない）
  • Claude Code はファイル変更前の状態をチェックポイントとして自動記録。/rewind でコードや会話を変更前に巻き戻せる
Safety netチェックポイントは便利な安全網ですが、コミットやブランチによる正式なバージョン管理（5章）の代わりにはなりません。
B

品質を高める

Verify, don't just trust

出力を鵜呑みにしないための仕組み。6章「AI出力との向き合い方」の、日々の実践面です。

モデルルーティング（軽量ファースト）

モデルルーティングは、タスクの難易度に応じて使うモデルを選び分けること。まず軽量モデルで試し、限界を感じたら上位へ切り替える「軽量ファースト」は、品質とコストの両立に有効です。

クロスレビューと反復

クロスレビューは複数AIで確認し合う手法。同じAIに観点を変えて反復させるのも効きます。

  • バグの調査をしてください
  • 前回と別の観点で調査をしてください
  • これまでの結果で、調べ足りない箇所がないか調査してください
  • これまでの結果の妥当性を検証し、優先度をつけてリスト化してください
Costクロスレビューや反復思考は効果的ですが、その分トークン（＝コスト）を多く消費します。使いどころを絞りましょう。

MCPで裏取りする

見覚えのないAPIや設定は、ハルシネーションの可能性があります。MCPで公式ドキュメントや最新仕様を参照させ、AI自身に裏取りさせてから採用します。

深く考えさせる（拡張思考）

難しい設計判断やバグ調査では、拡張思考（答える前に考える時間を長めに取る機能）が効きます。Claude Code では設定で有効/無効を切り替えられ、新しいモデルでは標準で有効です。あわせて「複数の案を比較してから答えて」「根拠も示して」のように促すと、検討の質がさらに上がります。

Trade-off深く考えるほど時間とトークンを使います。難所だけに使い、単純な作業では求めないのがコツ。テストを勝手に弱めさせない等の品質ルールは 5章 を参照。
C

スケールさせる

Scale the work

大きな仕事や複雑な入力に対応するための工夫。発展的なので、まず単一セッションに慣れてから少しずつ取り入れましょう。

ファイルと画像で渡す

  • 長大・複雑な内容はプロンプトに貼らず、ファイルに書いて読み込ませる
  • エラー画面やUIデザインは画像（スクリーンショット）で渡すと、言葉で説明するより速く正確に伝わる

サブエージェントで役割分担

大きなタスクはサブエージェントに切り出して役割分担させます。調査役・実装役・レビュー役のように分けると、見通しが良くなります。

Git worktree で並行作業

Git worktreeを使うと、1つのリポジトリから複数の作業場所を切り出し、別々のタスクを同時に進められます。作業どうしが干渉しないため、サブエージェントとの相性も良いです。

Note並行作業は強力ですが、その分管理も増えます。まず「1タスク1ブランチ」の基本に慣れてから。
D

コストを抑える

Spend fewer tokens

Claude Code は会話履歴や読み込んだファイルをすべてコンテキストとして扱うため、渡す情報が多いほどトークン（＝コスト）が増えます。必要なものだけを、必要なときに渡す——これが基本です。

対象を絞って渡す

最も効果が大きいのがこれ。修正を頼むときは、対象のファイルや関数を具体的に指定します。範囲を絞れば、関係ないファイルを読み込む無駄がなくなり、意図ズレも減ります。

✕ 曖昧「プロジェクト全体を見て、ログインのバグを直して」→ 関係ないファイルまで読み込み、トークンを浪費。修正範囲も広がりがち。
✓ 絞る「src/auth/login.ts のログイン処理だけを直して。関連するのは session.ts です」→ 必要な文脈だけを渡せて、速く・安く・正確に。
TipClaude Code では、プロンプトに @src/auth/login.ts のように @ファイルパス を書くだけで、そのファイルを対象として読み込ませられます。

コンテキストを整理する（/clear と /compact）

セッションが長いほど履歴が積み上がり、消費が増えます。/context で使用量を確かめながら、区切りで整理しましょう。

  • タスクが変わったら /clear … 履歴をリセット。消費削減の効果が最大
  • 会話が長くなりすぎたら /compact … 要約して引き継ぐ

定型化してキャッシュを効かせる

頻出プロンプトは CLAUDE.md や カスタムスラッシュコマンド に登録します。定型化した前提はキャッシュが効きやすく、同じセッション内での2回目以降のやり取りのコスト削減につながります（入力の手間が減る効果は、セッションをまたいでも有効です）。

In short「手戻りを減らすこと」も立派なコスト削減です。意図を固める（プラン承認）で無駄な生成を防ぎ、軽量ファーストで必要十分なモデルを選ぶ——これらはすべてトークン削減に効いてきます。
Cheat Sheet — Claude Code

コマンド早見表

よく使う Claude Code のコマンド。バージョンやツールにより挙動が異なる場合があります。

コマンド何をする？使いどころ
/initプロジェクト用の CLAUDE.md を自動生成最初の準備時
/clear会話履歴をリセットするタスクが変わったとき（消費削減大）
/compact会話を要約して引き継ぐ同じ作業を続けたいが履歴が重いとき
/rewindチェックポイントまで巻き戻す（コード・会話）AIの変更をやり直したいとき
/reviewPR をレビューさせる（手元差分は /code-review）変更内容を確認したいとき
/help使えるコマンドの一覧・使い方を表示するどんなコマンドがあるか忘れたとき
/model使うモデルを切り替えるタスクの難易度に応じて変えたいとき
/memoryCLAUDE.md などメモリファイルを編集するルールや前提を更新したいとき
/contextコンテキストの使用量を可視化する履歴が重くなっていないか確認したいとき
/permissionsツールの許可/拒否（権限）を管理する危険な操作を制限したいとき
